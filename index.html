How this tool works (the 3 bottlenecks)

Pre-training is compute-bound. Wall-clock time scales with total FLOPs C = 6·N·D (N params, D tokens; the 6 = 2 forward + 4 backward per param-token). The lever is model size.

Fine-tuning & RL are memory-bound. Full training holds ~16 bytes/param (weights + grads + FP32 Adam states). RL is worst: it keeps the policy, a frozen reference, an optional reward model, and a large rollout KV cache in memory at once.

Inference is bandwidth-bound. At batch 1, decode speed ≈ (bandwidth × util) / weight_bytes — every token must drag all weights across the 273 GB/s bus. Architecture (GQA/MLA/SSM) mainly shrinks the KV cache, not the weight traffic.

Open planning aid — order-of-magnitude estimates; validate with a short profiling run before a multi-day job. Compute uses C≈6·N·D; memory uses standard mixed-precision Adam accounting. Hardware figures: 128 GB unified memory, 273 GB/s bandwidth, BF16 tensor ~213 TFLOPS, FP4 dense ~427 TFLOPS (measured GB10, NVIDIA Developer Forums, Nov 2025). Not affiliated with NVIDIA.
